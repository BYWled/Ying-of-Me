一、前端到底在做什么？

前端开发，就是把“产品需求”变成用户在浏览器里看到并能操作的界面。

你每天点的按钮、看的卡片、输入框里的提示、页面上的动画，几乎都属于前端。

对应三件套：

  • HTML：页面结构（骨架）
  • CSS：页面样式（皮肤）
  • JavaScript：页面行为（神经系统）

五、核心原理

下面这部分是本文的“底层理解区”。

你可以把前端看成三层：

  1. HTML：定义信息结构与语义
  2. CSS：定义视觉规则与布局算法
  3. 浏览器渲染引擎：把前两者变成真实像素

5.1 HTML 深入： 不只是“标签拼接”

5.1.1 语义化

语义化的核心不是“好看”，而是让不同系统都能正确理解你的内容：

  • 浏览器：知道哪里是导航、正文、页脚
  • 搜索引擎：更容易抽取页面结构
  • 屏幕阅读器：能读出更准确的页面层次

常见语义标签（翻译即功能）：header、nav、main、section、article、aside、footer。

5.1.2 文档树（DOM）与节点关系

HTML 解析后不是字符串，而是树：

  • 父子关系：影响样式继承和选择器命中
  • 兄弟关系：影响相邻选择器与布局流
  • 节点顺序：影响键盘导航顺序与可访问性

5.1.3 块级 / 行内 / 行内块的本质

  • 块级：默认独占一行，参与普通文档流的块格式化
  • 行内：在文本行内排布，主要受行盒模型约束
  • 行内块：兼具“可同排 + 可设宽高”特性

这决定了“元素为什么换行 / 为什么宽高不生效”。

5.1.4 表单与可访问性基础

很多人只会写 <input>，但可用的表单还需要：

  • 合理设置 type（如 email、number）以获得原生校验与移动端键盘优化
  • 使用 button 的语义类型（submit、button）避免误提交
  • placeholder 是提示，不是标签本身

5.1.5 资源加载机制（脚本、样式、图片）

  • CSS 会影响首屏渲染，通常应尽早加载
  • JS 默认会阻塞解析（外链脚本尤其明显）
  • 图片天然懒解析，但会影响布局稳定性（需关注尺寸占位）

简化理解：

谁会阻塞“页面可见”，谁就优先优化。

5.1.6 为什么“结构正确”比“样式漂亮”更重要

当项目变大后，结构错误会导致：

  • 选择器越来越复杂
  • 脚本定位不稳定
  • 重构成本大幅上升

所以 HTML 最重要的是：层次清楚、语义准确、命名稳定。

5.2 CSS 深入：不只是“装饰语法”

5.2.1 层叠 —— CSS 的核心运行机制

最终样式由四件事共同决定：

  1. 来源（浏览器默认样式、作者样式、用户样式）
  2. 优先级（specificity）
  3. 书写顺序（后写覆盖先写）
  4. !important（强制提升优先权）

常见简化优先级认知：

!important > 内联样式 > #id > .class / [attr] / :pseudo-class > tag / ::pseudo-element

5.2.2 继承与初始值

并非所有属性都会继承：

  • 常继承：color、font-*、line-height
  • 常不继承：margin、padding、border、width、height

你会经常用到这些值：

  • inherit：继承父元素
  • initial：回到规范初始值
  • unset：可继承属性等价 inherit，不可继承属性等价 initial

5.2.3 盒模型与尺寸计算

元素真实占位通常由这套公式决定：

box-sizing: border-box 是现代开发默认选项，它让宽高计算更直觉。

5.2.4 格式化上下文（BFC / IFC）

这是很多布局“玄学问题”的根源：

  • BFC（块级格式化上下文）：可隔离浮动影响、避免外边距重叠干扰
  • IFC（行内格式化上下文）：文本和行内元素在同一行盒中布局

理解它后，你会明白为什么某些“清除浮动”技巧有效。

5.2.5 定位体系（position）

  • static：默认，按文档流
  • relative：保留原位置，再偏移自身
  • absolute：脱离文档流，相对最近定位祖先
  • fixed：相对视口固定
  • sticky：在阈值前后切换流内/固定行为

5.2.6 层叠上下文与 z-index

z-index 不是“值大就一定在最上面”，它受层叠上下文限制。

常见会创建新层叠上下文的场景：

  • 定位元素 + 非 auto 的 z-index
  • opacity < 1
  • transform / filter 等属性

这就是 z-index 写到 9999 仍盖不住的根因之一。

5.2.7 现代布局：Flex 与 Grid 的边界

  • Flex：一维布局（行或列），适合组件内部对齐
  • Grid：二维布局（行和列），适合页面级区域划分

常见策略：

  • 页面骨架用 Grid（或大块 Flex）
  • 组件内部用 Flex

如果你对布局有些了解，可能还听说过 CSS Float、Table、甚至定位布局等老方法。
这些方法在现代开发中已经很少使用了，建议先专注于 Flex 和 Grid。

5.2.8 响应式设计

核心不是“手机端单独写一套”，而是“同一套结构在不同视口渐进适配”：

  • 相对单位：%、em、rem、vw、vh
  • 媒体查询：@media (max-width: ...)
  • 流式布局：弹性宽度 + 自动换行

5.2.9 动画与性能

动画属性大致分两类：

  • 更容易触发重排：width、height、top、left 等
  • 更易合成加速：transform、opacity

经验法则：

动画尽量优先用 transform / opacity。

5.2.10 工程化 CSS 思维

当页面变大后，关键不是“能写样式”，而是“能维护样式”：

  • 命名体系（如 BEM 思路）
  • 组件级样式边界（避免全局污染）
  • 设计变量（颜色、间距、字号统一管理）
  • 减少过深选择器（降低耦合）

5.3 浏览器渲染与性能：从“写对”到“跑快”

5.3.1 渲染主流程（高频考点）

  1. HTML -> DOM
  2. CSS -> CSSOM
  3. DOM + CSSOM -> Render Tree
  4. Layout（几何计算）
  5. Paint（绘制）
  6. Composite（图层合成）

5.3.2 重排（Reflow）与重绘（Repaint）

  • 重排：元素几何信息变化（位置/尺寸）导致布局重新计算，成本更高
  • 重绘：仅视觉变化（如颜色）不改几何，通常成本较低

5.3.3 常见性能问题

  • 频繁读写布局信息（如循环里多次触发强制布局）
  • 大量复杂选择器导致样式计算压力上升
  • 图片无尺寸占位导致布局抖动

5.3.4 可执行优化清单（一般可直接用）

  • 优先简化 DOM 结构，减少无意义嵌套
  • 统一使用 border-box
  • 动画优先 transform / opacity
  • 图片提前给宽高，降低页面跳动
  • 样式分层组织，避免一处改动牵连全局

5.4 一套通用排错框架（HTML/CSS）

当页面“不对劲”时，按这个顺序排查：

  1. 先确认结构：标签闭合、层级、语义是否合理
  2. 再看选择器：是否选中目标元素
  3. 再看层叠：是否被更高优先级覆盖
  4. 再看布局模型：当前是普通流 / Flex / Grid / 定位哪一种
  5. 最后看渲染细节：是否有尺寸塌陷、图层遮挡、滚动容器影响

把改代码改为控制变量法：每次只改一个变量，再观察结果。
这是从初学者走向工程化开发的关键习惯。

下一步…

如果你已经读到这里，建议按这个节奏继续：

  1. 把本文示例手写 2 次（不要复制）
  2. 自己改 3 处样式（颜色、间距、卡片列数）
  3. 再加 1 个交互（比如“收藏”按钮）
  4. 学会用浏览器开发者工具(F12)看 DOM 和 CSS

当你能“改得动、讲得清”，前端入门就稳了。

前端不是记语法比赛，而是“结构 + 样式 + 交互 + 原理”的持续组合训练。
先做出来，再做规范，最后做性能。

关于AI的使用：

你可以用 AI 来辅助理解和查找资料，但不要直接复制 AI 生成的代码，要自己动手写一遍。

后期部分章节会介绍一些 AI 辅助工具的使用方法，帮助你更高效地学习和开发。

现在如果遇到问题，全然可以问 AI，但不要让 AI 替你完成练习，在 AI 给出解决方案后，务必自己动手实现一遍。
